# Day 06

Today was about not treating anomaly detection like a model-shopping exercise.

The useful question was not:
"which algorithm sounds more advanced?"

The useful question was:
"what kind of abnormal behavior am I trying to detect, and what information does each method actually use?"

## What I worked on

- reviewed the existing anomaly baseline already in the repo
- added statistical anomaly scores using `z-score` and `MAD`
- added a sequence-window pipeline for temporal anomaly detection
- added an LSTM autoencoder module for reconstruction-error based scoring
- set up a window-level comparison path so sequence modeling can be compared against simpler methods

## What this means in plain language

### `z-score`

`z-score` asks:
how far is this value from the normal mean, measured in standard deviations?

That makes it simple and easy to explain.

It works best when the data distribution is fairly stable and not too distorted by extreme values.

### `MAD`

`MAD` means median absolute deviation.

It asks:
how far is this value from the median, using a robust spread estimate instead of standard deviation?

This matters because sensor data can contain spikes and skew.
If I use only mean and standard deviation, a few extreme points can distort what "normal" looks like.

### LSTM autoencoder

The LSTM autoencoder is different because it does not judge one row in isolation.

It sees a sequence window.
It tries to compress that window into a smaller latent representation and then reconstruct the original sequence.

If it learned normal temporal behavior well, then:

- normal windows should reconstruct with low error
- unusual windows should reconstruct with higher error

That reconstruction error becomes the anomaly score.

## Why this matters in this project

This repo is not just tabular classification.

CMAPSS and the synthetic turbofan data are sequential:

- each `unit` is one engine
- each `cycle` is one time step in its life
- degradation is something that unfolds over time

That means some anomalies are not just extreme single values.
They can be odd temporal patterns:

- sudden drift
- unusual transitions
- instability over several cycles
- multivariate behavior that only looks strange across a window

That is the actual reason sequence models make sense here.
Not because LSTM is fashionable.

## Why I used it

I kept the statistical baselines because they are important reference points.

If a simple robust detector does almost as well as a deep model, that is a useful project finding.

I added the LSTM autoencoder because the project data is sequential and reconstruction error is a natural anomaly score when training mostly on normal behavior.

## What I learned

- a fair anomaly comparison should respect time structure, not just row-level scoring
- `z-score` and `MAD` are not weak just because they are simple
- the right deep-learning baseline here is a sequence autoencoder, not a random dense network on engineered rows
- thresholding matters just as much as model choice because the detector score still has to become a decision

## What the actual results said

### Row-level baseline experiment

On the row-level synthetic hold-out experiment:

- `Isolation Forest`: ROC-AUC `0.6398`, PR-AUC `0.1886`
- `Local Outlier Factor`: ROC-AUC `0.7335`, PR-AUC `0.5060`
- `Z-Score Distance`: ROC-AUC `0.7014`, PR-AUC `0.4832`
- `MAD Distance`: ROC-AUC `0.6993`, PR-AUC `0.4823`

What that says in plain language:

- `LOF` was the strongest classical detector in the row-level setup
- `z-score` and `MAD` were much stronger than I would expect if I only judged them by simplicity
- `Isolation Forest` was clearly the weakest of the four in this controlled setup

### Window-level comparison with the LSTM autoencoder

On the sequence-window comparison run:

- `MAD Distance`: ROC-AUC `0.941561`, PR-AUC `0.939986`
- `Z-Score Distance`: ROC-AUC `0.940698`, PR-AUC `0.939898`
- `LSTM Autoencoder`: ROC-AUC `0.933422`, PR-AUC `0.928510`
- `Local Outlier Factor`: ROC-AUC `0.918339`, PR-AUC `0.921313`
- `Isolation Forest`: ROC-AUC `0.759402`, PR-AUC `0.672790`

The LSTM autoencoder used:

- reconstruction-error threshold: `0.563681`
- training device: `cpu`
- final training loss: `0.408489`

What that says in plain language:

- the LSTM autoencoder worked well and learned useful temporal structure
- but the best methods in this synthetic setup were still the robust statistical baselines
- `MAD` and `z-score` were not just "fallback baselines"
- they were slightly better than the LSTM while being simpler and easier to explain

That is an important project result.
It means deep learning is justified only if it adds clear value beyond strong simple methods.

## What still feels shaky

- how sensitive reconstruction error will be to window length
- whether using all sensors is better than a curated subset
- whether thresholding from train reconstruction quantiles will generalize well enough
- how much synthetic anomaly injection matches realistic failure precursors

## Mistakes or traps

- using sequence models without a sequence-based evaluation setup
- comparing row-level and window-level metrics as if they mean the same thing
- treating reconstruction error as self-explanatory without checking threshold behavior
- assuming deep learning is automatically stronger than robust statistical baselines
