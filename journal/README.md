# Journal System

This folder is where the project gets translated into something you can actually work through.

The PDF gives scope.
This folder holds the working interpretation:

- what the plan really is
- what you learned
- what still feels unclear
- what the capstone eventually expects

## Files here

- `execution_timeline.md`
  The practical project plan.
  This is the one to read before starting a new block of work.

- `learning_journal.md`
  The main note-taking template.
  Use this when you need structure.

- `day_01.md` to `day_15.md`
  Daily working notes.
  These matter more than the template because they show what the project actually felt like while it was being built.

- `submission_checklist.md`
  A running checklist based on the PDF's deliverables and evaluation style.

- `project_pdf_extracted.txt`
  Raw text extracted from the original PDF for reference.

## How to use this folder without making it annoying

1. Read `execution_timeline.md` when you want direction.
2. Write in `day_0X.md` files when you want real notes.
3. Use `learning_journal.md` when you get stuck and need prompts.
4. Check `submission_checklist.md` once a week so the capstone requirements do not sneak up on you.

If the day files are good, someone should be able to read them and understand:

- what was built
- why a method was chosen
- what alternatives were considered
- what still felt uncertain at the end of the day

## What not to do

- do not turn the repo root README into a diary
- do not write fake polished reflections
- do not copy definitions you do not actually understand

If a note sounds impressive but you cannot explain it out loud, it is not helping.
